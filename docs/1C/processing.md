# Алгоритмы обработки CF-совместимых контейнеров

## Назначение документа

Документ описывает внутреннюю техническую спецификацию алгоритмов обработки CF-контейнеров (`.cf`, `.epf`, `.erf`) для плагина Far1c. Алгоритмы документируют формат данных и логику работы с ним, обеспечивая возможность независимой реализации.

## 1. Базовые служебные алгоритмы

### 1.1 Разбор пути

`parse_path(path)` разбивает путь на:

- каталог;
- имя файла без расширения;
- расширение.

Алгоритм построен на одном шаблоне сопоставления строки и используется для генерации имён выходных файлов:

```lua
function parse_path(path)
    return string.match(path, "(.-)([^\\/]-)%.?([^%.\\/]*)$")
end
```

### 1.2 Константы формата

Ключевые константы для работы с бинарным форматом:

| Константа | Значение | Назначение |
|-----------|----------|-------------|
| `SIG` | `0x7FFFFFFF` | Сигнатура конца цепочки страниц / сигнатура вложенного контейнера |
| `BOM` | `0xEF 0xBB 0xBF` | UTF-8 BOM (маркер несжатого текста) |
| `CRLF` | `0x0D 0x0A` | Разделитель строк в текстовых заголовках страниц |
| `ImageHeaderLen` | 16 байт | Размер заголовка образа (4 × uint32) |
| `PageHeaderLen` | 31 байт | Размер заголовка страницы (CRLF + 3×hex8 + CRLF) |
| `RowHeaderLen` | 20 байт | Размер заголовка строки (2×datetime + attributes) |

### 1.2 Создание множества

`new_set(list)` преобразует массив строк в хеш-множество вида `set[value] = true`. Этот приём дальше используется:

- для валидации расширений (`epf`, `erf`);
- для проверки GUID типов форм.

## 2. Низкоуровневые алгоритмы чтения контейнера

### 2.1 Абстракция Reader

Во всех алгоритмах чтения используется единый интерфейс:

- `read(len)`
- `setpos(pos)`
- `pos()`
- `size()`

Реализации две:

- `NewFileReader` для чтения из файла;
- `NewStringReader` для чтения из строки в памяти.

Это ключевой алгоритмический приём: вся логика контейнера работает не с файлом напрямую, а с абстрактным позиционным источником байтов.

### 2.2 Буферизованное файловое чтение

`NewFileReader(file, buflen)` реализует чтение большого файла без полной загрузки в память. Если размер буфера не передан явно, используется значение по умолчанию `256 * 1024` байт.

Алгоритм:

1. Вычислить размер файла.
2. Считать первый буфер фиксированного размера.
3. Держать две координаты:
   - `fpos` — позиция начала буфера в файле;
   - `offset` — смещение внутри буфера.
4. При `setpos(newpos)`:
   - вычислить новый `offset` через `newpos % buflen`;
   - переместить файловый курсор на начало нужного буфера;
   - перечитать буфер.
5. При `read(len)`:
   - дочитать хвост текущего буфера;
   - при необходимости переходить по полным буферам;
   - собрать результат конкатенацией фрагментов.

Смысл алгоритма: линейное и произвольное чтение контейнера работает на больших файлах с ограниченным расходом памяти.

### 2.3 Чтение из строки

`NewStringReader(s)` повторяет тот же интерфейс поверх строки в памяти. Он нужен для работы с уже распакованными вложенными контейнерами.

### 2.4 Парсер скобочной структуры

`NewStructReader(s)` разбирает текст формата:

```text
{value,{nested1,nested2},{"quoted,value"}}
```

Алгоритм:

1. Найти первую `{`.
2. Идти по строке по символам `["{},]`.
3. Поддерживать флаг `skip`, который показывает, находимся ли внутри кавычек.
4. При встрече `{` вне кавычек — рекурсивно разобрать вложенный узел.
5. При встрече `,` вне кавычек — завершить текущий лист.
6. При встрече `}` вне кавычек — завершить текущую ветку.
7. Вместо копирования строк хранить диапазоны `begin/end` листьев в массиве `leaf`.

Поверх дерева даются два алгоритма навигации:

- `goto(...)` — переход в подузел по числовому пути;
- `read(...)` — чтение листа по пути без смены контекста.

Это не универсальный parser combinator, а компактный специализированный разбор сериализованных метаданных.

### 2.5 Чтение little-endian `uint32`

`GetUInt32(b0, b1, b2, b3)` собирает 32-битное число из четырёх байтов в little-endian порядке. Алгоритм используется для бинарных полей заголовков и таблицы указателей.

## 3. Алгоритмы разбора физической структуры CF-контейнера

### 3.1 Формат уровней контейнера

Разбор организован по четырём сущностям:

- `ImageHeader`
- `PageHeader`
- `RowHeader`
- `RowBody`

#### `ImageHeader`

Читается первые 16 байт:

- сигнатура;
- размер страницы;
- ревизия;
- служебное поле.

#### `PageHeader`

Читается как текстовый блок фиксированной длины:

- проверяется обрамление `CRLF`;
- три поля интерпретируются как hex-строки:
  - `FullSize`
  - `PageSize`
  - `NextPage`

#### `RowHeader`

Читается из страницы-заголовка строки:

- timestamps и атрибуты пропускаются как второстепенные;
- идентификатор строки читается как остаток payload.

#### `RowBody`

Тело строки может занимать цепочку страниц. Алгоритм читает первую страницу, затем переходит по `NextPage`, пока не встретит сигнатуру конца цепочки.

### 3.2 Алгоритм чтения цепочки страниц

`ReadRowBody(rd)`:

1. Прочитать `PageHeader`.
2. Взять `min(FullSize, PageSize)` байт полезных данных.
3. Если `NextPage` не терминальный, перейти по адресу следующей страницы.
4. Повторять, пока не будет собран весь `FullSize`.
5. Проверить, что остаток равен нулю.

Алгоритм обеспечивает чтение как коротких, так и многостраничных блоков.

### 3.3 Алгоритм чтения области указателей

`ReadPages(rd, Pointers)` читает страницы, в которых лежат 32-битные указатели на строки.

Алгоритм:

1. Прочитать первый `PageHeader`.
2. На каждой странице прочитать массив 4-байтовых указателей.
3. Сохранить метаданные страницы в `Pages`.
4. Перейти на `NextPage`.
5. Последнюю страницу обработать частично, по остаточному `FullSize`.

Фактически это отдельная таблица индексов контейнера.

### 3.4 Алгоритм итерации по строкам контейнера

`ReadImage(rd)` создаёт объект изображения с итератором `Rows()`.

Ключевой алгоритм:

1. Считать заголовок изображения.
2. Считать страницы с указателями.
3. Обрабатывать указатели тройками:
   - адрес заголовка строки;
   - адрес тела строки;
   - терминатор `0x7FFFFFFF`.
4. Для каждой строки:
   - прочитать `RowHeader`;
   - удалить нулевые байты из `ID`;
   - прочитать `RowBody`;
   - определить признак `Packed`.

### 3.5 Эвристика определения сжатого блока

Блок считается несжатым, если тело начинается с UTF-8 BOM.

Иначе выставляется `Packed = true`.

Это не криптографическая проверка и не проверка формата DEFLATE, а прикладная эвристика, достаточная для конкретного контейнера.

## 4. Алгоритмы семантического разбора содержимого

### 4.1 Полная материализация контейнера в словарь

`ReadModulesFromFile(path)` строит отображение `dir[ID] = content`.

Алгоритм:

1. Открыть `.epf`/`.erf`.
2. Пройти все строки контейнера через `Rows()`.
3. Если блок помечен как сжатый:
   - выполнить inflate;
   - проверить первые 4 байта результата.
4. Если результат начинается с сигнатуры контейнера:
   - создать вложенный `Image`;
   - прочитать его строки в отдельный словарь;
   - сохранить словарь как значение `dir[ID]`.
5. Иначе сохранить распакованную строку.
6. Для несжатых блоков сохранить исходное тело.

Итоговая структура неоднородна:

- строка — для обычных данных;
- таблица/словарь — для вложенного контейнера.

### 4.2 Рекурсивное распознавание вложенных контейнеров

Проверка `res:sub(1, 4) == SIG` задаёт важный алгоритм классификации:

- сигнатура контейнера → вложенный контейнер;
- иначе → конечный payload.

Это позволяет строить дерево вложенности любой глубины без отдельной таблицы типов.

### 4.3 Навигация `root -> uuid -> content`

После сборки словаря выполняется навигация по метаданным:

1. Разобрать `dir["root"]` через `StructReader`.
2. Прочитать UUID корневого объекта: `root_head.read(2)`.
3. Разобрать содержимое `dir[root_uuid]`.
4. Дальше извлекать нужные узлы по жёстко заданным числовым путям.

Это алгоритм адресации метаданных без именованных полей: структура читается по позиционным индексам.

### 4.3.1 Точные числовые пути извлечения данных

Из корневого объекта метаданных:

| Данные | Путь | Описание |
|--------|------|----------|
| Имя объекта | `root_body.read(4, 2, 2, 4, 2, 3)` | Извлечение с обрезкой кавычек |
| UUID объекта | `root_body.read(4, 2, 2, 4, 2, 2, 3)` | Получение идентификатора объекта |
| Количество элементов | `root_body.read(3)` | Число дочерних элементов |
| Тип элемента | `root_body.read(i, 1)` | Идентификатор типа для проверки |

### 4.4 Извлечение модуля объекта

Алгоритм:

1. Из дерева метаданных прочитать имя объекта.
2. Прочитать UUID объекта.
3. Открыть содержимое `dir[uuid .. ".0"]`.
4. Если это вложенный контейнер, взять элемент `text`.
5. Добавить модуль в список как:
   - тип: `object`;
   - имя;
   - текст модуля.

### 4.5 Поиск форм по GUID типа

Используется множество двух GUID типов форм:

| GUID | Тип формы |
|------|-----------|
| `d5b0e5ed-256d-401c-9c36-f630cafd8a62` | Форма обработки |
| `a3b368c0-29e2-11d6-a3c7-0050bae0a776` | Форма отчёта |

Алгоритм:

1. Перебрать элементы корневой структуры.
2. Найти тот, чей тип входит в множество GUID форм.
3. Для найденного узла перебрать список UUID конкретных форм.

Таким способом список форм определяется не по имени ветки, а по типу объекта метаданных.

### 4.6 Извлечение имени формы

Для формы сначала разбирается её заголовок (`form_head`).

Дальше применяется ветвление по версии формата:

- если `form_head.read(2, 1) == "1"` — используется путь имени для формата 8.2: `form_head.read(2, 2, 2, 2, 3)`. Значение дополнительно очищается от кавычек.
- иначе используется альтернативный путь: `form_head.read(2, 2, 2, 3)`.

Также извлекается тип формы (управляемая или обычная) по пути: `form_type = form_head.read(2, 2, 2, 4)`.

Это алгоритм совместимости между вариантами внутреннего формата формы, определяющий точные позиционные индексы в скобочной структуре.

### 4.7 Извлечение модуля формы

Есть два алгоритма, зависящих от типа формы:

#### Управляемая форма

1. `form_type == "1"`.
2. Тело формы — строковая структура.
3. Она разбирается через `StructReader`.
4. Модуль берётся из поля `read(3)`.
5. Восстанавливается BOM и выполняется замена `""` на `"` внутри текста.

#### Обычная форма

1. Тело формы — вложенный контейнер.
2. Из него берётся файл `module`.

### 4.8 Итоговый алгоритм построения списка модулей

Возвращается массив записей:

- `mod_type`
- `mod_name`
- `mod_text`

В список включаются:

- модуль объекта;
- модули всех найденных форм.

## 5. Алгоритмы распаковки и перепаковки

### 5.1 Рекурсивная распаковка в каталог

Алгоритм реализует DFS-обход контейнера:

Алгоритм:

1. Прочитать контейнер.
2. Для каждой строки:
   - если блок сжат, выполнить inflate;
   - если результат — вложенный контейнер, создать каталог `ID/` и рекурсивно распаковать его;
   - иначе записать данные в файл `path + ID`.
3. Несжатые блоки сразу писать на диск.

Итогом становится файловое представление дерева контейнера.

### 5.2 Рекурсивная выгрузка в ZIP

Алгоритм повторяет обход, но вместо каталога формирует ZIP-архив:

Алгоритм:

1. Рекурсивно обойти контейнер.
2. Для вложенного контейнера спуститься глубже и накопить префикс пути.
3. Для конечных блоков записать данные в ZIP по пути `path + ID`.
4. Использовать быструю степень сжатия.

Это не сборка обратно в CF, а линейная выгрузка содержимого контейнера в ZIP-архив.

## 6. Алгоритмы извлечения и публикации модулей

### 6.1 Извлечение модулей в отдельные файлы

Базовый алгоритм выгрузки:

1. Прочитать список модулей через `ReadModulesFromFile`.
2. Для каждого модуля сформировать имя файла:
   - `<имя_обработки>_Модуль.txt` для модуля объекта;
   - `<имя_обработки>_<имя_формы>.txt` для модуля формы.
3. Сконвертировать имя в CP1251.
4. Сохранить текст модуля на диск.

Здесь есть алгоритм не только извлечения, но и нормализации имён файлов под Windows-окружение.

### 6.2 Автоматическое извлечение (pre-commit hook)

Пакетный алгоритм для синхронизации с VCS:

1. Читать список изменённых путей.
2. Для каждого существующего `.epf`/`.erf`:
   - извлечь модули;
   - записать `.txt` рядом с исходным файлом.
3. Если файл новый, выполнить `svn add`.

Это алгоритм синхронизации бинарных артефактов с текстовыми представлениями в VCS.

## 7. Алгоритмы интерактивного просмотра

### 7.1 Загрузка набора модулей в UI

Алгоритм инициализации интерфейса просмотра модулей:

1. Открыть файл обработки/отчёта.
2. Извлечь список модулей через `ReadModulesFromFile`.
3. Наполнить список имён модулей.
4. Показать текст первого модуля по умолчанию.

В оригинальной реализации используется GUI-фреймворк IUP. Для Far1c этот алгоритм адаптируется под панельный интерфейс Far Manager.

### 7.2 Поиск в тексте

Алгоритм поиска:

1. Взять текущую позицию каретки.
2. Если включён RegExp и есть выделение, скорректировать стартовую позицию.
3. Выполнить `find()` по тексту.
4. Если найдено:
   - переместить каретку;
   - выставить выделение;
   - прокрутить к позиции.
5. Если не найдено:
   - предложить поиск с начала;
   - при согласии сбросить каретку и повторить поиск.

Это циклический интерактивный поиск с двумя режимами:

- literal;
- regexp.

### 7.3 Переход к строке

Отдельный алгоритм позиционирования:

1. Запросить номер строки.
2. Установить `scrollto`.
3. Выделить диапазон строки.

## 8. Алгоритмы сравнения конфигурационных деревьев

### 8.1 Автоматическое определение кодировок

Перед разбором конфигурационного файла алгоритм выполняет эвристику:

1. Считывает первые 3 байта файла.
2. Если это `0xEF 0xBB 0xBF` (UTF-8 BOM), файл читается как UTF-8.
3. Иначе файл обрабатывается как ANSI (обычно Windows-1251 в контексте 1С).

При записи результата сравнения сохраняется исходная кодировка (включая запись BOM, если он был в оригинале).

### 8.2 Построение дерева по таб-отступам

Алгоритм читает текстовое представление (dump), где иерархия задаётся знаками табуляции.

Алгоритм:

1. Прочитать первую строку и определить базовый отступ.
2. Для каждой следующей строки:
   - вычислить текущий уровень отступа;
   - при углублении перейти в последний созданный узел;
   - при уменьшении отступа подниматься к родителю;
   - создать новую ветку.
3. На каждом уровне отсортировать дочерние узлы лексикографически (по полю `value`).

Значение узла очищается от служебного первого символа (`пробел` или `"`), если он присутствует.

Особенности реализации:
- Первый символ строки определяет способ обработки: пробел или кавычка означают, что значение заключено в кавычки.
- Дерево строится с сохранением ссылки на родителя (`parent`) для обратной навигации.
- Сортировка (`table.sort(tree.children, comp)`) выполняется после полного построения дерева для каждого уровня. Данная нормализация критически важна для корректной работы алгоритма сравнения.

### 8.3 Рекурсивное сравнение двух деревьев

`diff(t1, t2, indent, index, res)` работает как merge двух отсортированных списков детей (`children`).

Алгоритм:

1. Взять два указателя `i` и `j`.
2. Пока есть элементы хотя бы в одном списке:
   - если узел есть только справа (либо левый больше правого лексикографически) — пометить `-->`;
   - если узел есть только слева (либо правый больше левого) — пометить `<--`;
   - если значения равны — рекурсивно сравнить поддеревья.
3. Если в поддереве есть отличия, вывести и сам общий узел.
4. Если поддеревья идентичны, удалить временную запись из результата.

Это классический merge-diff для предварительно нормализованных (отсортированных) деревьев.

## 9. Алгоритмы компрессии и архивирования

### 9.1 DEFLATE Inflate

Алгоритм декомпрессии блоков DEFLATE:

1. Принять сжатый буфер.
2. Выполнить inflate (используется библиотека miniz/tinfl или эквивалент).
3. Вернуть распакованную строку или признак ошибки.

В реализации используется функция `tinfl_decompress_mem_to_heap` из библиотеки miniz.

### 9.2 Поток записи ZIP

Алгоритм ZIP writer:

1. `new_zip_writer(path)` — создать объект архива с файловым дескриптором.
2. `zip:write(internal_path, data, level)` — добавить запись в архив с заданным уровнем сжатия.
3. `zip:finalize()` — завершить архив и закрыть дескриптор.

Это служебный алгоритм упаковки для сценария экспорта, а не часть формата CF.

## 10. Выводы для собственной реализации

Документированные алгоритмы образуют следующие устойчивые блоки:

1. Абстрактный позиционный reader с буферизованным доступом.
2. Разбор физической структуры CF-контейнера через Image/Page/Row.
3. Итерация по строкам контейнера через таблицу указателей.
4. Эвристическое распознавание сжатых и вложенных блоков.
5. Парсер сериализованной скобочной структуры метаданных.
6. Навигация по метаданным через числовые пути.
7. Выделение модулей объекта и форм с учётом варианта формата.
8. Рекурсивная распаковка контейнера в дерево файлов.
9. Линейная выгрузка содержимого контейнера в ZIP.
10. Построение дерева по таб-структуре и рекурсивный diff.

Алгоритмы описаны как внутренняя спецификация и могут использоваться для независимой реализации в рамках проекта Far1c.
