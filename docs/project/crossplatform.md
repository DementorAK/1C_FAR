# **Исследование механизмов и совместимости API плагинов FAR Manager (v2, v3, far2l, far2m) для кроссплатформенной разработки**

## **Введение в архитектурную парадигму и историческую эволюцию FAR Manager**

Историческое развитие консольного файлового менеджера FAR Manager, первоначально созданного Евгением Рошалом в 1996 году 1, представляет собой уникальный пример эволюции системного программного обеспечения. На протяжении десятилетий FAR Manager удерживает позиции одного из наиболее гибких и расширяемых инструментов для работы с файловыми системами, архивами и системными ресурсами операционных систем.2 Фундаментальной основой такой долговечности и неизменной популярности среди системных администраторов и разработчиков является его открытая архитектура плагинов (Plugins API). Данная архитектура позволяет сторонним модулям глубоко интегрироваться в подсистемы графического интерфейса, встроенного редактора, просмотрщика и виртуальных файловых панелей.2 Начиная с версии 1.70 и последующего перехода к версии 2.0 в 2008 году, когда исходный код был открыт под модифицированной лицензией BSD 1, экосистема плагинов пережила бурный рост, предложив решения от FTP-клиентов до сложных сред выполнения скриптов.3

Смена поколений от версии 1.x к версии 2.x ознаменовалась критически важным переходом на полномасштабную поддержку стандарта Unicode.4 Данный переход потребовал существенной адаптации API и изменения базовых типов данных, так как внутреннее представление символов в операционной системе Windows базируется на кодировке UTF-16LE. Версия 3.x, развитие которой продолжила группа Far Group, повлекла за собой еще более радикальный пересмотр архитектурных контрактов. В стремлении к модернизации разработчики сознательно пошли на нарушение обратной совместимости (API break).4 Это было сделано ради внедрения современных концепций: повсеместного использования глобально уникальных идентификаторов (GUID/UUID) для идентификации плагинов и их компонентов, переноса хранилища конфигурационных настроек из системного реестра Windows в локальные транзакционные базы данных SQLite 4, а также значительного расширения структур данных для управления многообразием цветов (поддержка формата RGBA) и глубокой обработки асинхронных событий.5

Параллельно с эволюцией экосистемы в проприетарной среде Windows, возникла амбициозна инициатива портирования FAR Manager на POSIX-совместимые операционные системы, такие как GNU/Linux, macOS и различные деривативы BSD. Проект, получивший название far2l, инициированный энтузиастами сообщества (в частности, разработчиком elfmz) 7, представляет собой прямой форк исходного кода FAR Manager v2. Для обеспечения работоспособности Windows-ориентированного кода в Unix-средах был разработан собственный слой совместимости под названием WinPort. Этот слой динамически транслирует вызовы Win32 API в их POSIX-эквиваленты, обеспечивая абстракцию над файловыми дескрипторами, потоками исполнения и консольным вводом-выводом.7 В дальнейшем, на технологической базе far2l образовался независимый форк far2m, который интегрировал в себя продвинутую макросистему на базе языка LuaJIT (идеологически заимствованную из FAR 3\) и предложил расширенные функции API плагинов для более плотной интеграции с графическими оболочками Linux.7

В свете столь масштабной фрагментации платформ, инженерная задача создания единого, монолитного плагина, написанного на современном языке безопасного системного программирования Rust, который мог бы абсолютно единообразно функционировать во всех четырех целевых средах (Windows Far 2, Windows Far 3, Linux far2l, Linux far2m), представляет собой комплексную проблему проектирования архитектуры программного обеспечения. Разработчику необходимо преодолеть фундаментальные барьеры, связанные с различными контрактами бинарных вызовов (ABI), семантическими изменениями в сигнатурах экспортируемых функций, разной размерностью базовых символьных типов, таких как wchar\_t 8, а также несовместимыми парадигмами управления памятью при пересечении границы интерфейса внешних функций (Foreign Function Interface, FFI).10 Настоящий документ представляет собой исчерпывающее аналитическое исследование механизмов плагинов указанных версий и формулирует концептуальную архитектуру для реализации кроссплатформенного слоя абстракции (Safe Facade) на языке Rust, обеспечивающего универсальную совместимость без деградации производительности.

## **Сравнительный анализ архитектурных парадигм и бинарных интерфейсов**

Для грамотного проектирования универсального плагина требуется детальная декомпозиция анатомических различий между целевыми платформами. API плагинов FAR Manager строго регламентируется заголовочным файлом C++ plugin.hpp, который исторически служит незыблемым контрактом между диспетчером плагинов ядра и динамически загружаемой библиотекой (DLL в архитектуре Windows PE или SO в архитектуре Linux ELF).5 Анализ этого контракта сквозь призму версий выявляет существенные сдвиги в логике инициализации, идентификации и делегирования управления.

### **Среда выполнения FAR Manager 2 (Windows)**

Версия 2.x заложила современный фундамент Unicode-совместимых модулей, став стандартом де\-факто для множества корпоративных сред.4 Внутренняя архитектура FAR 2 строго привязана к парадигмам операционной системы Windows NT. В частности, сохранение пользовательских параметров, кэшей и путей плагинов жестко завязано на системный реестр, обычно по пути HKEY\_CURRENT\_USER\\Software\\Far2.4 Идентификация самих плагинов, их диалоговых окон и пунктов меню осуществляется на основе строковых констант и простых целочисленных индексов. Подобный подход, хотя и требовал минимальных накладных расходов, часто приводил к конфликтам имен при одновременной работе множества сторонних модулей.

Жизненный цикл плагина в версии 2 инициируется ядром сразу после проецирования динамической библиотеки в адресное пространство процесса. Диспетчер вызывает экспортируемую функцию SetStartupInfoW.12 Через эту функцию ядро передает плагину константный указатель на структуру PluginStartupInfo, которая содержит матрицу указателей на критически важные служебные функции ядра (Service functions), такие как Menu, Message, InputBox, AdvControl, а также указатель на структуру FarStandardFunctions (FSF), предоставляющую вспомогательные утилиты сортировки, копирования в буфер обмена и работы с путями.6 Чтобы предоставить ядру информацию о своих функциональных возможностях, плагин экспортирует функцию GetPluginInfoW, в рамках которой заполняется структура PluginInfo массивами строк для интеграции в главное меню, меню выбора дисков и меню конфигурации.11 Запуск активной фазы плагина инициируется вызовом функции OpenPluginW, возвращающей системный дескриптор HANDLE.5

### **Среда выполнения FAR Manager 3 (Windows)**

Релиз FAR Manager 3.0 ознаменовал собой архитектурный разрыв (API break), продиктованный необходимостью устранения технологического долга.4 Главным нововведением стал бескомпромиссный отказ от строковой идентификации в пользу использования 128-битных глобально уникальных идентификаторов (GUID/UUID).5 С этого момента каждое диалоговое окно, каждый пункт меню, каждая конфигурационная транзакция и сам плагин в целом требуют детерминированного уникального GUID.12 Настройки были полностью мигрированы из системного реестра в реляционные базы данных SQLite 4, что потребовало от разработчиков ядра создания совершенно нового набора API для работы с иерархической конфигурацией, известного как SettingsControl.15

Жизненный цикл подвергся пересмотру: первой вызываемой функцией стала GetGlobalInfoW, предоставляющая ядру базовые метаданные о версии плагина, минимально требуемой версии среды FAR Manager и, самое главное, глобальный GUID плагина.5 Функция SetStartupInfoW сохранилась, но состав структуры PluginStartupInfo был многократно расширен новыми интерфейсами. Функция запуска была окончательно стандартизирована и переименована в OpenW, а ее параметры были инкапсулированы в структуру OpenInfo, содержащую контекст вызова.5 Также были значительно переработаны структуры, описывающие диалоговые элементы (FarDialogItemData 6) и системы цветов. В версии 3 структура FarColor стала поддерживать как классические индексные цвета консоли, так и прямые значения TrueColor RGB/RGBA через механизмы C-совместимых объединений (unions) и битовых масок флагов (например, FCF\_FG\_BOLD, FCF\_BG\_ITALIC).5

### **POSIX-совместимый порт far2l (Linux, macOS, BSD)**

Проект far2l представляет собой колоссальную работу по переносу монолитной кодовой базы FAR 2 на Unix-подобные системы с соблюдением стандартов POSIX.7 Сохраняя общую логику API версии 2 (включая строковую идентификацию и отсутствие обязательных GUID), far2l вводит критическое архитектурное отличие на уровне абстракции компилятора: реализацию базовых типов данных символьного текста.8 В то время как Windows исторически применяет кодировку UTF-16, ядро Linux и стандартная библиотека libc (glibc/musl) используют спецификацию, где размер wchar\_t составляет 4 байта (кодировка UTF-32).8 Следовательно, размер всех структур в plugin.hpp, содержащих символьные данные или указатели на них, математически изменяется.

Для абстрагирования от платформозависимых системных вызовов разработчики far2l внедрили слой абстракции WinPort.8 Плагины для far2l компилируются как стандартные разделяемые библиотеки ELF (.so), однако внутри своего исходного кода они продолжают использовать макросы и типы данных из парадигмы Windows, такие как HANDLE, DWORD, WINAPI и функции вроде CreateFileW. Слой WinPort прозрачно транслирует эти вызовы в системные вызовы Linux. Важной и опасной для интеграции особенностью является то, что плагины не компонуются (не линкуются) с WinPort статически; все необходимые символы и трансляторы экспортируются в память самим исполняемым бинарным файлом far2l.8 Эта архитектура диктует строгие правила для разработчика на Rust, требуя точного совпадения сигнатур и недопустимости использования сторонних крейтов (подобных winapi), которые могут иметь конфликтующие определения типов.

### **Прогрессивный форк far2m (Linux)**

Форк far2m развивает идеологию far2l, интегрируя элементы, характерные для FAR 3, при сохранении базового API far2l.7 Наиболее заметным технологическим внедрением является интеграция подсистемы на базе компилятора LuaJIT, позволяющей выполнять высокопроизводительные скрипты Lua непосредственно в контексте внутреннего редактора, просмотрщика или файловых панелей.16 В контексте взаимодействия C/C++ API и плагинов, far2m добавляет расширенные функции в массив FarStandardFunctions. К таким функциям относятся Execute (предназначенная для выполнения команд в изолированном виртуальном терминале) и ExecuteLibrary (для исполнения символов загруженных разделяемых библиотек в отдельном процессе операционной системы).8

Кроме того, был добавлен специализированный API для системных уведомлений рабочего стола — DisplayNotification, интегрирующийся с демонами уведомлений Linux (через D-Bus или аналогичные шины).8 Введен макрос FCTL\_GETPANELPLUGINHANDLE, позволяющий одному плагину взаимодействовать с другим плагином, контролирующим активную панель.8 С точки зрения обратной совместимости, плагины, скомпилированные для far2l, как правило, функционируют в far2m без перекомпиляции, однако для полного использования новых возможностей асинхронного выполнения требуется адаптация слоя абстракции на языке Rust для перехвата макро-вызовов и передачи контекста в LuaJIT.8

## **Анализ структур данных и контрактов API подсистем**

Проектирование безопасных FFI-привязок (bindings) на языке Rust 10 требует исчерпывающего понимания не только сигнатур функций, но и способов распределения памяти, а также жизненного цикла структур данных, пересекающих границу модулей. Ниже приведен детальный разбор ключевых подсистем API, используемых во всех четырех целевых средах.

### **Инфраструктура виртуальных панелей (Panel API)**

Подсистема Panel API является сердцем файлового менеджера, позволяя плагинам эмулировать файловые системы (например, протоколы FTP/SFTP, просмотр содержимого архивов, системного реестра или баз данных).2 Жизненный цикл панели управляется набором строго регламентированных функций.

В FAR 2 и far2l/far2m ядро вызывает функцию GetFindDataW, передавая ей указатель на неинициализированный массив PluginPanelItem и переменную для записи количества элементов.13 Плагин обязан выделить память под этот массив (обычно используя функции распределения памяти из языка C, такие как malloc). В структуре PluginPanelItem содержится детальная метаинформация: атрибуты файла, времена создания/модификации/доступа, размер, а также флаги LIF\_PTRDATA, позволяющие ассоциировать с каждым файлом произвольные пользовательские данные через скрытый указатель.6 После того как ядро завершает отрисовку панели или закрывает ее, оно обязано вызвать экспортируемую плагином функцию FreeFindDataW 13, передав обратно указатель на массив, чтобы плагин мог освободить выделенную память, избегая утечек.

В FAR 3 парадигма Panel API была существенно расширена. Появилась функция предварительного анализа AnalyseW, которая принимает структуру AnalyseInfo и возвращает логическое значение (или HANDLE анализа), подтверждающее, способен ли плагин обработать переданный ему файл или поток данных.15 Это изменение позволило ядру более интеллектуально распределять файлы по плагинам архиваторов без необходимости полной инициализации инстанса плагина через OpenW. Структуры времени были переведены на использование 64-битных представлений FILETIME, а также добавлена поддержка расширенных полей контента через GetContentFieldsW и GetContentDataW.5

С точки зрения реализации на языке Rust, данная модель выделения и освобождения памяти представляет собой классическую проблему управления владением (Memory Ownership).10 Если плагин на Rust сформирует элементы панели в стандартном векторе Vec\<PluginPanelItem\> и передаст ядру указатель через метод .as\_mut\_ptr(), компилятор Rust автоматически деаллоцирует память вектора при выходе из текущей области видимости функции (scope). Это неминуемо приведет к уязвимости Use-After-Free на стороне C++ ядра FAR Manager. Архитектурно правильное решение заключается в конвертации безопасного вектора в "сырой" указатель через вызов Box::into\_raw или Vec::into\_raw\_parts. В последующем вызове FreeFindDataW этот сырой указатель должен быть безопасно возвращен под контроль Rust-аллокатора посредством Vec::from\_raw\_parts для корректного освобождения системных ресурсов.10

| Подсистема / Функция | Назначение в контексте виртуальной панели | Особенности реализации в Rust (FFI) |
| :---- | :---- | :---- |
| GetFindDataW | Генерация списка файлов эмулируемой ФС. | Требует перевода памяти Vec в unmanaged state через into\_raw\_parts. |
| FreeFindDataW | Освобождение памяти, выделенной в GetFindDataW. | Возврат управления памяти через from\_raw\_parts и неявный drop(). |
| PutFilesW | Передача файлов из хост-ФС в виртуальную панель. | Строгий парсинг путей; обеспечение catch\_unwind для I/O ошибок. |
| DeleteFilesW | Удаление файлов внутри плагина. | Обработка массива строк, парсинг кодировок UTF-16/UTF-32. |
| SetDirectoryW | Изменение текущего каталога внутри плагина. | Удержание состояния пути в Mutex или RwLock инстанса плагина. |

### **Механизмы ввода и оконные интерфейсы (Dialog API)**

Работа с интерактивными диалоговыми окнами представляет собой наиболее комплексную задачу FFI в контексте интеграции FAR Manager с языком Rust. API диалогов исторически требует передачи массива конфигурационных структур FarDialogItem 6 и указателя на функцию обратного вызова (Callback function) — DlgProc.18

Каждый элемент диалога (например, поле ввода текста DI\_EDIT, выпадающий список DI\_COMBOBOX, кнопка DI\_BUTTON или флажок DI\_CHECKBOX 5) описывается своими координатами X и Y, флагами состояний (такими как DIF\_VAREDIT или DIF\_HIDDEN 6) и указателем на данные FarDialogItemData.6 В FAR 3 система диалогов была переведена на архитектуру, управляемую сообщениями, аналогичную циклу обработки сообщений (Message Loop) в подсистеме Windows USER32. Функция обратного вызова DlgProc принимает параметры Msg (целочисленный код события), Param1 (индекс элемента) и Param2 (указатель на структуры данных, зависящие от типа сообщения, такие как нажатие клавиши мыши DN\_MOUSECLICK или изменение фокуса DN\_LISTCHANGE 6). Плагин может активно вмешиваться в отрисовку, отправляя сообщения ядру через служебную функцию SendDlgMessage.14

Проектирование безопасной архитектуры для диалогов в Rust подразумевает полный отказ от ручного манипулирования массивами памяти. Требуется внедрение паттерна "Builder" (Строитель), который позволяет разработчику конструировать форму декларативно:

Rust

let mut dialog \= DialogBuilder::new(guid)  
   .title("Конфигурация сетевого протокола")  
   .add\_checkbox("Использовать кэширование соединений", true)  
   .add\_text\_input("Таймаут (сек)", "30")  
   .add\_button("Применить", DefaultButton::Yes)  
   .build();

При вызове метода .show(), слой абстракции (Facade) автоматически конвертирует эту безопасную конфигурацию в низкоуровневый массив FarDialogItem, выделяет необходимую память на куче для хранения строк, и регистрирует небезопасную C-совместимую функцию обратного вызова (имеющую атрибут extern "system"). Когда ядро FAR Manager инициирует вызов этой DlgProc в ответ на действия пользователя, обертка FFI извлекает контекст Rust (используя указатель Param или глобальный хэш-словарь дескрипторов), маршрутизирует событие в безопасный обработчик, предоставляемый пользователем плагина, и возвращает корректный код ответа ядру, предварительно обеспечив перехват любых возможных паник через catch\_unwind.19

### **Редактор и Просмотрщик (Editor & Viewer API)**

FAR Manager славится своим высокопроизводительным встроенным редактором и средством просмотра файлов.20 Плагины могут перехватывать управление этими подсистемами посредством функций ProcessEditorInputW, ProcessEditorEventW и ProcessViewerEventW.13

Функция ProcessEditorInputW позволяет плагину выступать в роли фильтра нажатий клавиатуры или макро-перехватчика. Ядро передает плагину структуру INPUT\_RECORD (в FAR 3 инкапсулированную в FAR\_INPUT\_RECORD с полем IntKey 5), и если плагин возвращает ненулевое значение, ядро считает событие обработанным и не передает его базовому редактору.13 Это позволяет создавать плагины для автодополнения кода, переформатирования текста и интеграции с внешними линтерами (например, Colorer).6 Функция ProcessEditorEventW информирует плагин о событиях жизненного цикла, таких как EE\_READ (файл загружен в память), EE\_SAVE (пользователь сохранил изменения) или EE\_REDRAW (требуется перерисовка экрана).6 Для активного управления текстом плагин использует служебный вызов EditorControl 14, через который можно внедрять цветовые атрибуты (ECTL\_ADDCOLOR), манипулировать закладками (ECTL\_GETBOOKMARKS) и позицией курсора.6

В контексте Linux-форков far2l и far2m, подсистема просмотра и редактирования осложнена поддержкой графических терминалов. Far2l поддерживает режимы отображения через GUI (библиотека WX) или расширения протокола X11/Xi для консолей (TTY), что позволяет перехватывать сложные манипуляции мышью и богатые цветовые схемы.7 Разработчик на Rust должен учитывать, что коды клавиш и модификаторы мыши, приходящие через слой WinPort, могут отличаться семантически в зависимости от запущенного бекенда TTY или GUI.

## **Вектор совместимости: Архитектурные барьеры кроссплатформенной разработки**

Реализация единой бинарной или исходной кодовой базы на Rust для столь различных операционных сред требует решения фундаментальных системных проблем. Данные проблемы выходят далеко за рамки тривиального условного преобразования типов и затрагивают глубинные аспекты семантики языка C/C++ в противовес строгим правилам компилятора Rust.10

### **1\. Фундаментальная проблема размерности и кодировки wchar\_t**

Символьный тип данных wchar\_t был исторически внедрен в стандарт языка C для представления "широких" символов, однако комитет по стандартизации намеренно не определил его точную битовую размерность, оставив это на усмотрение реализаций компиляторов.9 На платформе Microsoft Windows все ведущие компиляторы (включая MSVC и порты MinGW) строго фиксируют размер wchar\_t равным 2 байтам, что семантически соответствует кодировке символов UTF-16LE.24 API FAR Manager версий 2 и 3 для Windows неявно и повсеместно полагаются на этот факт: все строковые параметры, передаваемые в функции (имена файлов, элементы списков, конфигурационные ключи), представляют собой массивы 16-битных целых чисел, обязательно завершающиеся нулевым символом (Null-terminated strings).

В среде операционных систем семейства Linux (и, как следствие, в форках far2l и far2m) стандарт System V Application Binary Interface (ABI) и базовая библиотека языка C (glibc) определяют размер типа wchar\_t как 4 байта (32 бита), что строго соответствует универсальной кодировке UTF-32LE.8

Для экосистемы Rust этот дуализм создает критический архитектурный разрыв. В языке Rust стандартный скалярный тип char всегда гарантированно имеет размер 4 байта и представляет собой валидное значение Unicode.25 При этом базовый строковый тип выделяемой памяти String (и его строковый срез \&str) всегда использует компактную кодировку UTF-8.25

Следствия для кроссплатформенного слоя абстракции (FFI-биндингов):

1. **При компиляции под Windows-target:** Инфраструктура FFI должна транслировать итерации Rust \&str в структуру Vec\<u16\> с использованием стандартного метода .encode\_utf16(), обязательно добавляя терминальный нуль в конец буфера перед передачей указателя C++ коду.  
2. **При компиляции под Linux-target (far2l/far2m):** Инфраструктура FFI должна выполнять итерацию по символам \&str, приводя каждый char к типу u32 (или платформозависимому libc::wchar\_t), формируя Vec\<u32\> и также добавляя 32-битный терминальный нуль.8  
3. **Реверсивное преобразование:** Чтение строк, возвращаемых от ядра FAR Manager (например, путей директорий при вызове GetFindDataW), требует применения функции String::from\_utf16 в Windows и безопасного алгоритмического парсинга UTF-32 массивов в Linux. Учитывая, что вызовы API, связанные с отрисовкой панелей, происходят тысячи раз в секунду при активном скроллинге списка файлов, неоптимизированные строковые аллокации могут стать фатальным узким местом (bottleneck) производительности. Эффективная реализация должна опираться на пулы буферов (Buffer pooling) и избегать излишних копирований памяти там, где возможно заимствование.

### **2\. Динамическая линковка и парадигма слоя WinPort**

Слой WinPort, используемый в проектах far2l и far2m, представляет собой уникальный, но весьма рискованный инженерный компромисс.8 Вместо колоссальной задачи полного переписывания миллионов строк легаси-кода FAR Manager для прямого использования POSIX API, разработчики реализовали "прокладку", предоставляющую такие фундаментальные типы Windows как HANDLE, DWORD, WINAPI и эмулирующую функции наподобие CreateFileW, FindFirstFileW в среде Linux.8

Однако для плагина, разрабатываемого на языке Rust, это генерирует парадоксальную ситуацию в системе сборки. При осуществлении кросс-компиляции плагина под целевую платформу x86\_64-unknown-linux-gnu, компилятор Rust (rustc) и пакетный менеджер Cargo закономерно отключают доступ к системным крейтам Windows API (таким как winapi или windows-sys 26), так как они предназначены исключительно для Windows-таргетов. Если разработчик попытается определить типы наподобие HANDLE вручную, он должен гарантировать их стопроцентное бинарное совпадение с определениями из заголовочных файлов WinPort. Любое микро-отклонение (например, неверное определение размера HANDLE как u32 вместо беззнакового указателя \*mut c\_void на 64-битной платформе) приведет к скрытым повреждениям памяти (Memory Corruption) в рантайме, что крайне сложно поддается отладке.10

Дополнительную сложность вносит макрос WINAPI.8 В экосистеме Windows этот макрос обычно разворачивается компилятором MSVC в конвенцию вызова \_\_stdcall на 32-битных системах, или сливается с \_\_cdecl в архитектуре x64. В операционной системе Linux макрос WINAPI в контексте заголовочных файлов WinPort может быть определен как пустое значение, оставляя стандартный для платформы контракт вызовов C (System V AMD64 ABI). Это означает, что экспорт функций на Rust должен оформляться как extern "system" для корректной работы в Windows и extern "C" для интеграции в Linux.10 Достижение этого требует виртуозного применения директив условной компиляции \#\[cfg(...)\] непосредственно на уровне сигнатур экспортируемых макросов.

### **3\. Механизм StructSize, выравнивание памяти (Memory Alignment) и ABI**

Весь API FAR Manager опирается на классический паттерн версионирования структур через первое поле, именуемое StructSize. Каждая конфигурационная или информационная структура (например, GlobalInfo, OpenInfo, PluginInfo) начинается с поля, определяющего ее точный размер в байтах.1 Ядро FAR Manager использует значение этого поля для детерминирования версии API, с которой был скомпилирован плагин. Это позволяет ядру безопасно обрабатывать структуры, которые могут быть искусственно увеличены (в новых версиях) или уменьшены (в старых плагинах), предотвращая чтение за границами выделенной памяти.

В языках C/C++ инициализация данного поля выглядит тривиально и безопасно: Info-\>StructSize \= sizeof(\*Info).1 Однако в Rust разработчику необходимо математически точно гарантировать соответствие бинарной раскладки структуры (memory layout) ожиданиям C++ компилятора, которым было собрано ядро FAR. Применение атрибута \#\[repr(C)\] к структурам Rust является категорически обязательным требованием для отключения внутренних оптимизаций компилятора Rust, которые могут произвольно изменять порядок полей для минимизации паддинга (padding).10 Вызов встроенной функции std::mem::size\_of::\<GlobalInfo\>() обеспечит корректное вычисление размера, однако на платформе Linux (из-за 4-байтового wchar\_t и других фундаментальных отличий слоя WinPort) размер абсолютно идентичной по смыслу структуры будет кардинально отличаться от ее Windows-аналога. В случае малейшей ошибки в выравнивании (например, из\-за игнорирования директивы \#pragma pack в C++, которая не была должным образом отражена в Rust коде), ядро FAR Manager неминуемо отклонит плагин при попытке инициализации, расценив переданное значение StructSize как некорректное или потенциально опасное.1

### **4\. Несовместимость механизмов перехвата исключений (Panic Unwinding)**

Ядро FAR Manager, написанное на языках C/C++, не интегрировано с механизмами раскручивания стека вызовов (stack unwinding), инициируемого механизмом паник в коде, скомпилированном на Rust. В случае, если какая-либо функция плагина, синхронно вызванная из ядра FAR, вызовет макрос panic\!() (например, при выходе за пределы массива или ошибке разыменования Option::unwrap()), и процесс раскручивания стека пересечет границу FFI, это приведет к классическому состоянию неопределенного поведения (Undefined Behavior).10 В большинстве современных операционных систем это мгновенно выражается в аварийном завершении всего приложения файлового менеджера с потерей несохраненных данных пользователя (Crash / SIGILL / SIGABRT).

Для радикального устранения этой угрозы, каждая без исключения экспортируемая FFI-функция плагина обязана быть инкапсулирована в блок std::panic::catch\_unwind.19 Этот механизм гарантирует, что любая паника внутри Rust-кода будет остановлена до возврата управления в C++ ядро. При перехвате паники плагин должен корректно деаллоцировать локальные ресурсы, опционально зарегистрировать ошибку в лог-файл (или вывести через Message API) и вернуть ядру заранее оговоренное спецификацией API значение, символизирующее отказ (например, FALSE, 0 или NULL).

## **Разработка универсального плагина на Rust: Стратегии проектирования и Фасад**

Учитывая вышеперечисленные критические барьеры, архитектура кроссплатформенного плагина на Rust ни в коем случае не должна базироваться на наивном, построчном портировании C-заголовочных файлов в структуры данных. Требуется внедрение многоуровневой архитектурной стратегии, которая физически разделяет низкоуровневые системные FFI-привязки (Unsafe Bindings Layer) и высокоуровневый безопасный слой абстракции (Safe Facade Layer).10

### **Этап 1: Автоматизированная генерация биндингов и конфигурация сборки**

Для того чтобы система сборки Cargo сгенерировала динамическую библиотеку, полностью совместимую с механизмом загрузки плагинов FAR Manager (которая ожидает файл с расширением .dll в Windows или .so в Linux без специфичного для компилятора Rust манглирования символов имен), в корневом конфигурационном файле Cargo.toml необходимо явно определить правильный тип артефакта 27:

Ini, TOML

\[lib\]  
crate-type \= \["cdylib"\]

Этот параметр является директивой для компилятора rustc сгенерировать разделяемую библиотеку, соответствующую стандартам целевой платформы (C ABI). Это позволяет диспетчеру плагинов ядра FAR прозрачно загружать модуль в оперативную память посредством встроенных функций операционной системы (например, LoadLibraryW в экосистеме Win32 или dlopen в POSIX-системах).28

Автоматизированная генерация FFI-привязок (bindings) из исходных файлов plugin.hpp является предпочтительным методом, реализуемым с помощью утилиты bindgen.26 Однако, из\-за фундаментальных макро-различий между FAR v2, FAR v3 и Linux-форками far2l/far2m, использование одного универсального прохода bindgen технически нереализуемо. Разработчику необходимо задействовать скрипт сборки build.rs, который будет динамически выполнять оркестрацию:

1. Анализировать целевую архитектуру и операционную систему (target\_os, target\_arch).  
2. Читать активированные флаги конфигурации Cargo (features), такие как far3, far2, far2l или far2m.  
3. Динамически применять bindgen к соответствующей версии заголовочного файла plugin.hpp, предварительно внедряя в процесс парсинга необходимые макросы (например, \-D\_WIN32 для эмуляции Windows-окружения или специфичные флаги включения WinPort для эмуляции Linux-окружения).

### **Этап 2: Проектирование безопасного Фасада (Safe Traits)**

Надежный кроссплатформенный плагин на Rust должен максимально абстрагировать бизнес-логику от "сырых" FFI-указателей и C-совместимых структур. Слой Фасада представляет собой набор Rust Traits (типажей), описывающих функциональность плагина в строгом, типобезопасном и идиоматическом стиле.

Концептуальное определение основного трейта плагина:

Rust

pub trait FarPlugin {  
    /// Возвращает глобальные метаданные. Адаптер вызовет её только для v3.  
    fn get\_global\_info(&self) \-\> PluginGlobalInfo;  
      
    /// Возвращает конфигурацию меню. Адаптер сам трансформирует строки в wchar\_t.  
    fn get\_plugin\_info(&self) \-\> PluginMenuInfo;  
      
    /// Основная функция запуска (из командной строки, меню, ассоциированного файла)  
    fn open(&mut self, context: OpenContext) \-\> Result\<PanelHandle, PluginError\>;  
      
    /// Генерация содержимого виртуальной панели  
    fn get\_find\_data(&mut self, panel: PanelHandle) \-\> Result\<Vec\<PanelItem\>, PluginError\>;  
      
    /// Обработка событий встроенного редактора  
    fn process\_editor\_input(&mut self, event: EditorInputEvent) \-\> bool;  
}

Модель "Обертки" (Adapter/Wrapper) в слое FFI связывает этот безопасный Trait с небезопасным C API, экспортируемым плагином. Рассмотрим пример вызова GetPluginInfoW в среде FAR 2 или far2l. Функция-обертка extern "system" fn GetPluginInfoW перехватывает управление, блокирует мьютекс глобального инстанса плагина, вызывает метод plugin.get\_plugin\_info(), конвертирует возвращенные безопасные структуры PluginMenuInfo (содержащие стандартные Rust String) в выделенные на куче нуль-терминированные массивы wchar\_t. Затем обертка сохраняет эти сырые указатели в глобальном реестре аллокаций (для предотвращения их сборки мусором) и аккуратно передает их в поля структуры PluginInfo, предоставляемой ядром.11

Для версии FAR 3 адаптер функции GetGlobalInfoW вызовет plugin.get\_global\_info() и безопасно заполнит поля C-структуры GlobalInfo, сгенерировав байтовые массивы для структур GUID из безопасных высокоуровневых представлений UUID языка Rust.5

### **Этап 3: Абстрагирование API Конфигурации (SettingsControl vs Registry)**

Как было установлено при анализе платформ, FAR 2 сохраняет настройки плагинов в иерархическом системном реестре Windows 4, FAR 3 использует транзакционные локальные базы данных SQLite 4, а проекты far2l/far2m применяют собственные механизмы сериализации (чаще всего INI-подобные файлы) в директории \~/.config/far2l/settings/.31

Слой абстракции (Facade) должен герметично изолировать основную бизнес-логику плагина от этих технологических деталей реализации. Плагину должен инжектироваться интерфейс доступа к хранилищу конфигурации, например, SettingsStore:

Rust

pub trait SettingsStore {  
    fn set\_string(&mut self, key: &str, value: &str) \-\> Result\<(), ConfigError\>;  
    fn get\_string(&self, key: &str) \-\> Result\<Option\<String\>, ConfigError\>;  
    fn set\_int(&mut self, key: &str, value: i64) \-\> Result\<(), ConfigError\>;  
}

Реализация данного типажа будет жестко зависеть от целевой платформы и управляться макросами условной компиляции (\#\[cfg(...)\]). При сборке под конфигурацию FAR 3, структура-адаптер будет делегировать вызовы низкоуровневому API SettingsControl ядра FAR 15, инициируя транзакции в SQLite-хранилище. В конфигурации FAR 2 обертка будет использовать системный крейт winreg для прямого, но безопасного взаимодействия с кустами реестра Windows. Для платформы far2l будет реализован парсер текстовых конфигурационных файлов.

### **Этап 4: Интеграция с подсистемами макросов (LuaJIT в far2m и FAR 3\)**

FAR 3 и его функциональный Linux-наследник far2m глубоко интегрируют высокопроизводительную систему макросов на базе движка LuaJIT.7 Для разработчика кроссплатформенного плагина на Rust это открывает две критические возможности:

1. Вызов макросов непосредственно из кода плагина для оркестрации сложными манипуляциями средой FAR Manager (например, управление окнами других плагинов, извлечение выделенного текста).  
2. Предоставление собственных функций-обработчиков плагина в глобальную область видимости для вызова их из пользовательских Lua-скриптов.

Глубокая интеграция с подсистемой MacroControl 15 позволяет плагину отправлять сложные макро-последовательности в движок FAR. Использование структуры ActlKeyMacro и системного вызова MCMD\_POSTMACROSTRING 6 позволяет программно автоматизировать рутинные задачи, создавая уровень взаимодействия, выходящий за рамки традиционного управления панелями. Фасад на Rust должен предоставлять методы для абстрагирования от структуры FarMacroCall, обеспечивая строгую типобезопасную передачу и возврат параметров (целочисленных массивов, строк, булевых флагов) между диспетчером плагина на Rust и LuaJIT-контекстом виртуальной машины ядра.

В специфичном случае операционной системы Linux и платформы far2m, разработчик плагина на Rust может задействовать уникальные для этой ветки функции из FarStandardFunctions, такие как Execute (исполнение bash/shell команд в эмулированном виртуальном терминале с поддержкой pty) и ExecuteLibrary (динамическая загрузка SO-библиотек).8 Кроме того, вызов DisplayNotification позволяет плагину отправлять системные D-Bus уведомления в среду рабочего стола Linux (GNOME, KDE), информируя пользователя о завершении длительных фоновых операций плагина (например, скачивания файлов по FTP).8 Доступ к этим функциям в кодовой базе Rust должен быть обернут в атрибуты условной компиляции \#\[cfg(feature \= "far2m")\], чтобы гарантировать, что попытка вызова отсутствующих символов в Windows не приведет к падению.

## **Стратегия компиляции, непрерывной интеграции и дистрибуции (CI/CD)**

Для обеспечения заявленной тотальной совместимости с версиями 2/3 и far2l/far2m, инфраструктура сборки (билд-система) Rust-проекта должна использовать многовекторную матрицу кросс-компиляции.32 В идеальной архитектуре процесс непрерывной интеграции (CI, например, в GitHub Actions) должен автоматически собирать и публиковать следующие бинарные артефакты (targets):

1. **Сборка Windows FAR 3:** Целевая архитектура x86\_64-pc-windows-msvc с активированной опцией Cargo (feature) far3. Эта сборка создаст .dll, использующую подсистему GUID, SQLite настройки и обновленную структуру FarColor. Она предназначена для инсталляций современных 64-битных версий FAR Manager 3\.4  
2. **Сборка Windows FAR 2 (Legacy):** Целевые архитектуры i686-pc-windows-msvc (32-бит) и x86\_64-pc-windows-msvc с опцией far2. Генерирует устаревшую, но все еще востребованную сборку, использующую строковые идентификаторы, классическую структуру цветов и реестр Windows. Совместима с базой инсталляций классического FAR Manager 2\.4  
3. **Сборка Linux far2l / far2m:** Целевая архитектура x86\_64-unknown-linux-gnu с опциями far2l или far2m. Процесс сборки создаст разделяемый объектный файл .so. Исходный код динамически переключится на использование 32-битного типа wchar\_t и макросов, совместимых с прослойкой WinPort.8 Результирующий бинарный файл будет напрямую загружаться и функционировать в сборках far2l в большинстве современных дистрибутивов Linux (Debian, Ubuntu, Arch) без дополнительной компиляции на стороне пользователя.7  
4. **Сборка macOS / BSD (far2l):** Целевые платформы aarch64-apple-darwin и x86\_64-apple-darwin для обеспечения работы плагина в порте far2l под операционную систему Apple macOS.7

Такой бескомпромиссный модульный подход к компиляции полностью нивелирует классическую проблему фрагментации и "зоопарка версий" на уровне конечного потребителя. Пользователь просто загружает бинарный артефакт, строго соответствующий его операционной системе и версии файлового менеджера, в то время как инженер-разработчик поддерживает единую, монолитную и логически консистентную кодовую базу на современном языке Rust. Разрешение всех FFI-конфликтов, преобразований типов, выравниваний памяти и выбора кодировок строк элегантно делегируется слою абстракции Фасада (Safe Facade) на этапе автоматизированной компиляции.

## **Заключение**

Интеграция современного языка системного программирования Rust, обладающего уникальной моделью безопасности памяти и строгой системой типов, в многолетнюю и исторически унаследованную экосистему C/C++ плагинов FAR Manager — это амбициозная задача, требующая исключительной архитектурной дисциплины и глубокого понимания системных парадигм. Детальный ретроспективный анализ API наглядно демонстрирует, что технологическая эволюция от консервативной версии 2 к модернизированной версии 3 на платформе Windows, а также появление и стремительное развитие POSIX-форков far2l и far2m, сформировали высокофрагментированную вычислительную среду. Радикальные изменения в механизмах постоянной конфигурации (Системный реестр против локальной базы SQLite), системах уникальной идентификации (Символьные строки против бинарных GUID), и, что наиболее критично, в фундаментальных типах данных символьного представления (размерность wchar\_t в 2 или 4 байта при интеграции слоя WinPort), делают категорически невозможным использование прямых, неадаптированных биндингов исходного C/C++ API.

Тем не менее, проведенное техническое исследование безоговорочно доказывает, что проектирование и создание единого, единообразно и надежно функционирующего кроссплатформенного плагина на языке Rust абсолютно осуществимо на практике. Стратегия успеха базируется на бескомпромиссном и строгом применении фундаментальных парадигм языка Rust: тотальной изоляции небезопасного кода (взаимодействие FFI и блоки unsafe) в специализированных платформозависимых адаптерах, безусловном перехвате любых возможных паник на границе вызовов (через catch\_unwind) для обеспечения абсолютной стабильности диспетчера ядра FAR, а также на использовании продвинутого паттерна проектирования Фасад (Safe Facade). Именно Фасад берет на себя ответственность за абстрагирование тяжеловесных системных вызовов — открытие многоэлементных диалоговых окон, генерацию содержимого виртуальных панелей, динамический вызов Lua-макросов — от чистой бизнес-логики разрабатываемого плагина.

Внедрение и развитие подобной архитектуры не только обеспечит бесшовную поддержку широчайшего спектра операционных систем (от различных версий Microsoft Windows до дистрибутивов GNU/Linux, macOS и FreeBSD) и всех актуальных веток развития FAR Manager (v2, v3, far2l, far2m). Прежде всего, это концептуально гарантирует высочайший, ранее недостижимый уровень стабильности и надежности. Достигается это за счет строгих проверок этапа компиляции, алгебраической системы типов и уникального механизма заимствований (Borrow Checker) языка Rust, которые алгоритмически предотвращают классические и наиболее опасные ошибки ручного управления памятью (такие как Use-After-Free, Double Free, переполнения буферов и утечки дескрипторов ресурсов), исторически преследовавшие разработку системных плагинов на C/C++. Данный передовой вектор архитектурной разработки представляет собой не просто академический интерес, а наиболее оптимальный и прагматичный путь для качественной модернизации, стабилизации и безграничного расширения функциональности консольных интерфейсов файловых менеджеров в реалиях современных, высоконагруженных и гетерогенных гибридных вычислительных сред.
