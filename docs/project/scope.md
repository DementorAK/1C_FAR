# Project Scope Statement: Мультиплатформенный плагин FAR Manager для артефактов 1С

## 1. Обоснование проекта (Project Justification)

Разработчики и администраторы 1С, предпочитающие консольные инструменты (FAR Manager, far2l/far2m), нуждаются в нативном и кроссплатформенном способе низкоуровневой работы с артефактами платформы «1С:Предприятие 8». Существующие решения либо устарели, либо недоступны на Linux. Разработка плагина обеспечит удобную работу с внутренностями 1С-файлов напрямую из файлового менеджера без использования самой платформы 1С.

## 2. Цели проекта (Project Objectives)

Создать кроссплатформенный плагин для файлового менеджера, позволяющий прозрачно (подобно архивам) просматривать, извлекать и модифицировать структуру и содержимое бинарных файлов 1С в ОС Windows и Linux.

## 3. Описание продукта (Product Scope)

Плагин реализован на языке **Rust** с **C-совместимым интерфейсом** (C ABI) для корректной работы с Far Plugin API. Rust обеспечивает кроссплатформенную компиляцию, безопасную работу с памятью и нативную производительность.

**Поддерживаемые целевые платформы:**

* Windows: FAR Manager 3
* Linux: far2m (планируется базовая совместимость архитектуры для возможного портирования на far2l)

**Поддерживаемые форматы (полный цикл Чтение/Запись):**

* Файловые базы данных (`.1cd`) — просмотр таблиц, извлечение и замена вложенных конфигураций `.cf`.
* Конфигурации и расширения (`.cf`, `.cfe`).
* Внешние обработки и отчеты (`.epf`, `.erf`).

## 4. Границы проекта (Project Boundaries)

### 4.1. Включается в проект (In Scope)

* **Работа как с архивами:** Открытие файлов по Ctrl+PgDn/Enter с отображением внутренних сущностей в виде виртуальной файловой системы.
* **Модификация (Read-Write):** Распаковка (извлечение) элементов контейнера, их редактирование и обратная пересборка (упаковка) модифицированного артефакта.
* **Работа с .1cd:** Замещение внутренних файлов конфигураций (`.cf`) в базе внешними файлами без использования конфигуратора.
* **Защищенные модули:** Извлечение и отображение модулей без исходных кодов с расшифровкой в виде ассемблеро-подобных инструкций (опкодов платформы).

### 4.2. Исключается из проекта (Out of Scope)

* Интеграция с внешними CI/CD и dev-инструментами (Docker, Allure, vrunner, OneScript). Вся работа ограничена экосистемой Far Manager.
* Обфускация/деобфускация структуры кода (кроме показа готового байт-кода).
* Полноценный запуск кода платформы 1С.

## 5. Ограничения проекта (Constraints)

* **Работа с большими файлами:** Архитектура плагина должна изначально поддерживать работу с огромными конфигурациями и базами данных (реализация частичного чтения, ленивой загрузки «lazy-loading», ограничение потребления RAM).
* **Пользовательский опыт (UX):** Все ресурсоемкие и длительные операции (например, перепаковка контейнеров) обязательно должны сопровождаться индикатором выполнения (прогресс-баром), исключая субъективное «зависание» интерфейса FAR.

## 6. Допущения (Assumptions)

* **Внешние зависимости:** Допускается полагаться на наличие в системе внешних инструментов (например, 7-zip или библиотек сжатия), так как они уже включены в стандартную поставку FAR Manager в составе плагина ArcLite.
* **Декомпрессия DEFLATE:** Плагин не использует собственные FFI-обёртки над C-кодом (miniz, zlib). Вся (де)компрессия делегируется стандартным архивным плагинам FAR Manager (ArcLite / 7-Zip).
* **Основа парсинга:** Алгоритмы парсинга CF-контейнера, вложенных блоков и сериализованных метаданных описаны во внутренней спецификации [processing.md](../1C/processing.md) и реализуются в кодовой базе проекта.

## 7. Критерии приемки (Acceptance Criteria)

Продукт будет считаться готовым к релизу при успешном прохождении следующих пользовательских сценариев:

1. **Редактирование обработок и отчетов:** Пользователь открывает файл `.epf`/`.erf` как архив, изменяет текст модуля (`.bsl`), макета или XML-описания формы во встроенном редакторе (F4). При сохранении изменений плагин пересобирает контейнер, используя UX и горячие клавиши, привычные по работе с ArcLite. Модифицированный артефакт успешно принимается платформой 1С.
2. **Вложенная каскадная сборка:** Пользователь редактирует модуль встроенной обработки `.epf`, которая физически находится внутри конфигурации `.cf`. При сохранении изменений, плагин должен сначала переупаковать саму обработку, а затем автоматически обновить её внутри родительского `.cf`-контейнера.
3. **Редактирование защищенных модулей:** Открытие защищенного модуля (`.bsl`), отображение его внутренностей в виде читаемых ассемблеро-подобных инструкций, возможность редактировать эти команды и корректная пересборка (трансляция обратно в байт-код) в контейнер.
4. **Замещение в файловых базах:** Вход внутрь базы `.1cd` как в директорию, копирование из нее файла конфигурации (`.cf`), а также копирование в базу внешнего `.cf`-файла в режиме замещения (обновление конфигурации в базе напрямую).
5. **Управление составом контейнера:** Возможность добавления, изменения и удаления целых элементов-вложений внутри конфигурации или расширения (`.cf`/`.cfe`) — например, внешних отчетов, макетов, обработок, реквизитов справочников.
