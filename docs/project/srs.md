# Software Requirements Specification: 1C Far Plugin

**Версия:** 2.0  
**Статус:** Draft  
**Дата:** 2026-04-27

---

## 1. Введение

### 1.1 Наименование системы

**Название:** `far1c`  
**Полное название:** 1C Far Plugin  
**Описание:** Мультиплатформенный плагин для FAR Manager / far2m

### 1.2 Назначение системы

Плагин обеспечивает прозрачную работу с бинарными артефактами платформы 1С:Предприятие 8 как с виртуальной файловой системой. Пользователь работает с артефактами так же, как с обычными файлами и папками FAR Manager: просматривает внутреннюю структуру, редактирует содержимое, копирует и удаляет элементы. Артефакты на диске (`.epf`, `.erf`, `.cf`, `.cfe`, `.1cd`) при этом остаются обычными файлами — к ним применимы все стандартные файловые операции FAR.

### 1.3 Ключевой принцип UX

> Пользователь нажимает **Enter** на артефакте — FAR «заходит внутрь» как в директорию. Заголовок панели меняется. Всё остальное — привычные операции FAR.

Артефакты на файловой панели ведут себя как обычные файлы (перемещение F6, переименование Shift+F6, удаление F8, копирование F5 — всё работает на уровне файловой системы). Enter/Ctrl+PgDn открывает виртуальную панель внутри артефакта.

### 1.4 Целевая аудитория

- **Программисты 1С** — просмотр и доработка конфигураций, обработок, отчётов
- **Внедренцы** — работа на удалённых серверах через SSH/Terminal
- **DevOps / vibe-coders** — извлечение кода для AI-анализа, Git-интеграции

### 1.5 Язык реализации

Плагин реализуется на **Rust** с **C-совместимым интерфейсом** (C ABI) для корректной работы с Far Plugin API. Rust обеспечивает:

- Кроссплатформенную компиляцию (Windows / Linux)
- Безопасную работу с памятью без GC
- Нативную производительность
- Богатую экосистему crate'ов для работы с бинарными форматами и сжатием

### 1.6 Регламент версионирования

Версия плагина имеет формат `Major.Minor.Patch.Build`:

- **Major (Релиз):** `0` во время разработки, `1` после завершения всех фаз [плана работ](plan.md).
- **Minor (Фаза):** Номер текущей Фазы (например, `1` для Фазы 1, `2` для Фазы 2).
- **Patch (Задача):** Номер текущей задачи внутри Фазы (например, `6` для задачи 1.6).
- **Build (Сборка):** Автоинкремент значения при каждой новой сборке. Значение сбрасывается в `1` при смене номера задачи, Minor или Major.

## 2. Поддерживаемые форматы

### 2.1 Матрица форматов

| Формат | Расширение | Описание | R/W | Фаза |
|--------|------------|----------|-----|------|
| Внешняя обработка | `.epf` | Контейнер с модулями и формами | R/W | Фаза 1 (MVP) |
| Внешний отчёт | `.erf` | Контейнер с модулями и макетами | R/W | Фаза 1 (MVP) |
| Конфигурация | `.cf` | Основная конфигурация базы | R/W | Фаза 2 |
| Расширение | `.cfe` | Расширение конфигурации | R/W | Фаза 2 |
| Файловая база | `.1cd` | База данных 1С (с вложенными .cf/.cfe) | R/W | Фаза 3 |

### 2.2 Архитектурная иерархия артефактов

```
.1cd (Файловая база данных)
├── [ИмяКонфигурации].Config.cf       -- Рабочая конфигурация (всегда одна)
├── [ИмяПоставщика].Vendor.cf         -- Конфигурации поставщика (0..N штук)
├── DataBase.cf                        -- Конфигурация БД (может не совпадать с Config)
└── [ИмяРасширения].cfe               -- Расширения (0..N штук, включая неактивные)

.cf / .cfe (Конфигурация / Расширение)
├── CommonModules/
│   └── [Имя]/
│       └── Module.bsl
├── Documents/
│   └── [Имя]/
│       ├── ObjectModule.bsl
│       ├── ManagerModule.bsl
│       ├── Forms/
│       │   └── [ИмяФормы]/
│       │       └── Module.bsl
│       ├── Templates/
│       ├── Commands/
│       └── ...
├── Catalogs/   Reports/   DataProcessors/   ...  (аналогично Documents)
└── ... (все типы объектов метаданных)

.epf / .erf (Внешняя обработка / Внешний отчёт)
├── ObjectModule.bsl                   -- Модуль объекта
├── Forms/
│   └── [ИмяФормы]/
│       └── Module.bsl
├── Templates/
│   └── [ИмяМакета]
└── Commands/
    └── [ИмяКоманды]
```

### 2.3 Типы содержимого внутри контейнеров

| Тип | Описание | Формат в виртуальной ФС |
|-----|----------|--------------------------|
| Модуль | Код на языке 1С/BSL | Файл `.bsl` |
| Форма | Описание интерфейса | Папка с `Module.bsl` и описанием |
| Макет | Печатная форма | Файл (`.mxl`, `.html`, и др.) |
| Картинка | Изображение | Файл (`.png`, `.bmp`) |
| Защищённый модуль | Модуль без исходного кода | Файл `.bsl` (содержимое — дизассемблированный байт-код) |

## 3. Функциональные требования

### 3.1 Регистрация плагина

#### FR-001: Регистрация в системе

```
Требование: Плагин регистрируется в FAR Manager и доступен через F11.
Экспортируемые функции Far Plugin API (C ABI):
  - SetStartupInfoW(Info)
  - GetPluginInfoW(Info)
  - OpenW(Info)
  - AnalyseW(Info)
Префикс команды: "1c:"
Обрабатываемые расширения: .epf, .erf, .cf, .cfe, .1cd
```

#### FR-002: Стандартные файловые операции на артефактах

```
Требование: Файлы .epf/.erf/.cf/.cfe/.1cd на файловой панели FAR
ведут себя как обычные файлы.
Поддерживаются без участия плагина:
  - F5  -- копирование файла
  - F6  -- перемещение/переименование файла
  - F8  -- удаление файла
  - Shift+F6 -- переименование файла
Плагин вступает в работу только при Enter / Ctrl+PgDn.
```

#### FR-003: Активация виртуальной панели

```
Требование: Enter или Ctrl+PgDn на артефакте открывает
виртуальную панель внутри него.
Заголовок панели формируется по схеме:
  "FAR 1C:<ТИП>:<ИмяФайла>"
Примеры:
  "FAR 1C:1CD:test.1cd"
  "FAR 1C:CF:test.cf"
  "FAR 1C:CFE:test.cfe"
  "FAR 1C:EPF:test.epf"
  "FAR 1C:ERF:test.erf"
Backspace / Ctrl+PgUp -- выход из виртуальной панели.
```

### 3.2 Виртуальная файловая система

#### FR-010: Панель .1cd (Файловая база)

```
Требование: При входе в .1cd панель показывает вложенные
конфигурации и расширения как файлы .cf и .cfe.

Отображаемые элементы:
  [ИмяКонфигурации].Config.cf
    -- Рабочая конфигурация. Всегда присутствует.
    -- Имя берётся из метаданных конфигурации.

  [ИмяПоставщика].Vendor.cf
    -- Конфигурация(и) поставщика. Может быть 0..N штук.
    -- Каждая отображается отдельным файлом с уникальным именем.

  DataBase.cf
    -- Конфигурация базы данных. Всегда присутствует.
    -- Может отличаться от Config.cf (если конфигурация
       ещё не применена к базе).

  [ИмяРасширения].cfe
    -- Расширения. Может быть 0..N штук.
    -- Показываются все расширения, в том числе неактивные.
    -- Каждое отображается отдельным файлом.

Колонки: Имя, Размер, Тип
Тип: "Рабочая конфигурация" / "Конф. поставщика" /
     "Конф. БД" / "Расширение (активно)" / "Расширение (неактивно)"
```

#### FR-011: Панель .cf / .cfe (Конфигурация / Расширение)

```
Требование: При входе в .cf или .cfe панель показывает
иерархию объектов метаданных в виде папок и файлов.

Верхний уровень — типы объектов (папки):
  CommonModules/, Documents/, Catalogs/, Reports/,
  DataProcessors/, InformationRegisters/, AccumulationRegisters/,
  ExchangePlans/, Subsystems/, Roles/, ... и т.д.

Вложенный уровень — объект (папка):
  Documents/SaleOrder/
    ObjectModule.bsl       -- Модуль объекта
    ManagerModule.bsl      -- Модуль менеджера
    Forms/
      DocumentForm/
        Module.bsl         -- Модуль формы
      ListForm/
        Module.bsl
    Templates/
      PrintTemplate        -- Макет печатной формы
    Commands/
      PostCommand
    ...

Модули показываются как файлы .bsl непосредственно в папке объекта,
а не как вложенные папки — для удобства навигации.
```

#### FR-012: Панель .epf / .erf (Обработка / Отчёт)

```
Требование: При входе в .epf или .erf панель показывает
внутреннее содержимое обработки/отчёта.

Структура панели:
  ObjectModule.bsl           -- Модуль объекта
  Forms/
    [ИмяФормы]/
      Module.bsl             -- Модуль формы
  Templates/
    [ИмяМакета]              -- Макет
  Commands/
    [ИмяКоманды]             -- Команды

Внутри .cf/.cfe обработки и отчёты отображаются как папки
с тем же содержимым (не как бинарные файлы).
Это обеспечивает каскадный вход без промежуточного шага.
```

#### FR-013: Навигация по иерархии

```
Требование: Пользователь перемещается по иерархии артефакта
стандартными средствами FAR.
Поддерживаемые операции:
  - Enter / Ctrl+PgDn -- вход в папку или артефакт
  - Backspace / Ctrl+PgUp -- возврат на уровень выше
  - SetDirectory("..") -- выход
  - SetDirectory("FolderName") -- вход в подпапку
  - Выделение нескольких элементов (Insert, *, +, -)
Заголовок панели обновляется при каждом переходе.
```

#### FR-014: Вход из .1cd в конфигурацию

```
Требование: При нажатии Enter на файле .cf или .cfe внутри
панели .1cd плагин прозрачно извлекает конфигурацию во
временную директорию и открывает её виртуальную панель.

Поведение:
  1. Временный .cf извлекается в системный TEMP
  2. Открывается виртуальная панель с заголовком
     "FAR 1C:CF:[ИмяКонфигурации].Config.cf"
  3. Временный файл удаляется при закрытии панели
  4. Пользователь не знает о временном файле

Аналогично работает для .cfe.
```

### 3.3 Операции с файлами внутри виртуальной панели

#### FR-020: Просмотр (F3)

```
Требование: Просмотр содержимого элемента во встроенном viewer FAR.
Для текстовых файлов (.bsl, .xml):
  - Открыть FAR viewer с подсветкой синтаксиса (через Colorer)
Для бинарных файлов (макеты, картинки):
  - HEX-просмотр
Для защищённых модулей:
  - Отображение дизассемблированного листинга BSL-инструкций
```

#### FR-021: Редактирование (F4)

```
Требование: Редактирование элемента во встроенном редакторе FAR
с сохранением изменений обратно в контейнер.
Для текстовых файлов (.bsl):
  - Извлечь во временный файл, открыть редактор
  - При закрытии с сохранением (F2) — пересборка контейнера
  - Создание резервной копии с временной меткой перед изменением
    Формат имени: [имя].[YYYYMMDD-HHMMSS].[расширение]
    Пример: test.20260512-143022.epf
  - Создание бэкапа контролируется настройкой плагина
Для защищённых модулей (опционально, Фаза 6):
  - Редактирование дизассемблированного кода
  - Обратная трансляция в байт-код при сохранении
```

#### FR-022: Копирование из контейнера (F5)

```
Требование: Извлечение элементов из контейнера на файловую панель.
Поддержка:
  - Один файл или папка
  - Несколько выделенных элементов
  - Диалог "Extract to..." с выбором пути
```

#### FR-023: Копирование в контейнер (F5 / F6 с другой панели)

```
Требование: Добавление или замещение файлов в контейнере.
Поддержка:
  - Копирование .bsl файла поверх существующего модуля
  - Копирование .cf/.cfe файла в .1cd (замещение конфигурации)
  - Обновление индексов и перепаковка контейнера
```

#### FR-024: Удаление (F8)

```
Требование: Удаление элементов из контейнера с пересборкой.
Клавиши: F8, Shift+Del
Подтверждение: Диалог с перечнем удаляемых элементов
Ограничение: Корневые конфигурации в .1cd удалять нельзя
```

### 3.4 Распаковка

#### FR-030: Режимы распаковки

```
Требование: Поддержка различных режимов извлечения контейнера целиком.
Вызов: Shift+F2 или через меню
Режимы:
  1. V8Unpack-style -- файлы .header + .data (для совместимости)
  2. Raw -- бинарные блобы
  3. Full-parse -- структурированные файлы (.bsl, .xml, и др.)
  4. Saby-style -- читаемая структура для Git
```

#### FR-031: Распаковка в Git-пригодную структуру (Saby-style)

```
Требование: Извлечение конфигурации в плоскую структуру файлов,
удобную для хранения в Git и diff.
Структура вывода (пример):
  ConfigName/
  ├── Documents/
  │   └── SaleOrder/
  │       ├── ObjectModule.bsl
  │       ├── ManagerModule.bsl
  │       └── Forms/
  │           └── DocumentForm/
  │               └── Module.bsl
  ├── CommonModules/
  │   └── CommonModule1/
  │       └── Module.bsl
  └── index.json    -- Индекс всех объектов конфигурации
```

### 3.5 Упаковка

#### FR-040: Пересборка контейнера

```
Требование: Обратная сборка контейнера при сохранении изменений.
Операции:
  - Автоопределение версии формата
  - Выбор уровня сжатия
  - Сохранение метаданных
  - Резервное копирование оригинала с временной меткой
    Формат: [имя].[YYYYMMDD-HHMMSS].[расширение]
    Настраивается через диалог настроек плагина
Индикация: Прогресс-бар при операциях > 1 сек
```

#### FR-041: Каскадная пересборка

```
Требование: При сохранении изменённого вложенного контейнера
автоматически обновляется родительский.
Пример (EPF внутри CF):
  1. Пользователь редактирует .bsl внутри обработки в .cf
  2. При сохранении: пересобирается внутренний блок EPF
  3. Затем: обновляется родительский .cf
Пример (CF внутри 1CD):
  1. Пользователь копирует в .1cd новый .cf
  2. В базе замещается соответствующая конфигурация
```

### 3.6 Работа с .1cd

#### FR-050: Извлечение конфигурации из базы

```
Требование: Копирование .cf из .1cd на файловую панель (F5).
F5 на файле "MyConfig.Config.cf" → файл "MyConfig.Config.cf" на диске
```

#### FR-051: Замещение конфигурации в базе

```
Требование: Запись .cf файла с диска в .1cd с замещением.
F5/F6 (копирование в панель .1cd) → выбор целевой конфигурации → обновление базы
```

#### FR-052: Сравнение конфигураций

```
Требование: Визуальное сравнение двух конфигураций из одной базы.
Пример: Config.cf vs DataBase.cf -- выявление неприменённых изменений
```

### 3.7 Защищённые модули

#### FR-060: Обнаружение защищённых модулей

```
Требование: Определение и индикация защищённых модулей в панели.
Признаки: отсутствие текстового контента, наличие байт-кода.
Индикация: специальный значок/суффикс в имени файла, например [protected]
```

#### FR-061: Дизассемблирование

```
Требование: Отображение байт-кода защищённого модуля в читаемом виде.
Формат: ассемблеро-подобные инструкции BSL-опкодов платформы.
Навигация: процедуры, функции, метки переходов.
```

#### FR-062: Редактирование байт-кода (Фаза 4, опционально для MVP)

```
Требование: Изменение инструкций с обратной трансляцией в байт-код.
Операции: редактирование инструкций, валидация, сборка.
```

## 4. Требования к пользовательскому интерфейсу

### 4.1 Заголовок панели

#### UI-001: Схема заголовка

```
Формат: "FAR 1C:<ТИП>:<путь/имя файла>"
Типы:
  1CD  -- для файловых баз .1cd
  CF   -- для конфигураций .cf
  CFE  -- для расширений .cfe
  EPF  -- для внешних обработок .epf
  ERF  -- для внешних отчётов .erf
Примеры:
  "FAR 1C:1CD:test.1cd"
  "FAR 1C:CF:MyConfig.Config.cf"   (при входе из .1cd)
  "FAR 1C:CF:test.cf"              (при прямом открытии)
При навигации внутрь подпапки заголовок не меняется
(меняется только путь в строке состояния).
```

### 4.2 Прогресс-бар

#### UI-010: Индикация выполнения

```
Требование: Все операции > 1 секунды отображают прогресс.
Элементы:
  - Название операции
  - Текущий обрабатываемый элемент
  - N из M / процент выполнения
  - Кнопка отмены (Esc)
```

### 4.3 Диалоги

#### UI-020: Диалог распаковки

```
Опции:
  - Режим (V8Unpack / Raw / Full-parse / Saby-style)
  - Целевая папка
  - Перезаписывать существующие (да/нет)
```

#### UI-021: Диалог упаковки

```
Опции:
  - Уровень сжатия (минимальный / оптимальный / максимальный)
  - Создать резервную копию (да/нет)
  - Версия формата (авто / 8.2 / 8.3)
```

#### UI-022: Диалог сравнения

```
Опции:
  - Левая конфигурация (из панели .1cd)
  - Правая конфигурация (из панели .1cd)
  - Режим (быстрый / полный)
```

### 4.4 Сообщения об ошибках

#### UI-030: Формат ошибок


### 5.1 Потребление памяти

|-------------|-----------|

| 100 МБ – 500 МБ | 100 МБ |

| Операция | Время |
|----------|-------|

| Открытие панели (< 500 МБ) | < 5 сек |
| Открытие панели (> 500 МБ) | < 30 сек + прогресс |
| Распаковка 100 МБ | < 10 сек |

### 5.3 Lazy loading

```
Требование: Для файлов > 100 МБ реализовать ленивую загрузку.
Поведение:
  1. Показать верхний уровень сразу (по индексу из заголовка CF)
  2. Не читать содержимое блоков до обращения к ним
  3. Кешировать прочитанные блоки в памяти
```

## 10. Архитектура реализации

### 10.1 Принятые архитектурные решения

**ADR-001: Нативный плагин FAR на Rust с C ABI**

Плагин реализуется как нативная динамическая библиотека (`.dll` на Windows, `.so` на Linux) на Rust, экспортирующая функции Far Plugin API через `extern "C"`. Это обеспечивает:

- Прямой доступ к Far Plugin API без промежуточных слоёв
- Нативную производительность парсинга бинарных форматов
- Безопасную работу с памятью (Rust ownership model)
- Единую кодовую базу для Windows и Linux

**ADR-002: Структура проекта**

Архитектура проекта разделена на три независимых логических слоя:

```
src/
├── lib.rs                       -- Точка входа, экспорт Far Plugin API (C ABI)
├── far/                         -- СЛОЙ 1: Взаимодействие с Far Manager
│   ├── mod.rs                   -- GUID-ы плагина, глобальное состояние
│   ├── api.rs                   -- Полные биндинги Far Plugin SDK 3.0 (C ABI)
│   ├── panels.rs                -- Логика виртуальных файловых панелей (VFS, навигация, commit)
│   ├── ui.rs                    -- Отрисовка элементов UI (диалоги, прогресс-бары, меню)
│   ├── lang.rs                  -- Локализация: чтение .lng-файлов через Far API
│   └── settings.rs              -- Настройки плагина (стиль распаковки, бэкап)
├── v8/                          -- СЛОЙ 2: Семантика и структура артефактов 1С
│   ├── mod.rs
│   ├── container.rs             -- Чтение CF-контейнеров (ImageHeader, rows, страницы)
│   ├── vfs_builder.rs           -- Построение VFS-дерева из рядов контейнера
│   ├── writer.rs                -- CF-writer: пересборка контейнеров
│   ├── uuids.rs                 -- UUID-ы типов объектов метаданных 1С
│   └── tests.rs                 -- Интеграционные тесты парсинга и репаковки
└── base/                        -- СЛОЙ 3: Низкоуровневые библиотечные функции (I/O, парсинг)
    ├── mod.rs
    ├── reader.rs                -- Абстрактный ридер (FileReader, StringReader)
    ├── parser.rs                -- Низкоуровневый парсер скобочных структур 1С
    └── deflate.rs               -- DEFLATE (де)компрессия через flate2
```

**ADR-003: Диспетчеризация по типу файла**

`AnalyseW()` определяет тип по расширению → `OpenW()` выбирает панельный модуль:

- `.1cd` → `panels::onecd`
- `.cf`, `.cfe`, `.epf`, `.erf` → `panels::artifact`

**ADR-004: Временные файлы при входе из .1cd**

При входе в конфигурацию внутри .1cd: файл извлекается во временную директорию системы, открывается как обычный `.cf`, удаляется при закрытии панели. Пользователь не замечает этого.

**ADR-005: Декомпрессия DEFLATE через flate2**

Вместо отдельной FFI-обёртки над miniz используется Rust crate `flate2` для нативной (де)компрессии DEFLATE. Это устраняет зависимость от внешних архивных плагинов FAR и обеспечивает:

- Нативную производительность без overhead
- Кроссплатформенность из коробки
- Отсутствие внешних зависимостей для базовой функциональности

**ADR-006: Портирование алгоритмов из внутренней спецификации CF-обработки**

Алгоритмы парсинга CF-контейнера из `cf_reader.lua` и семантического разбора из `cf_inside.lua` берутся как референсная реализация и переписываются на Rust:

- Общая логика парсинга CF сохраняется, DEFLATE реализуется через `flate2`
- Семантический разбор расширяется для поддержки `.cf`/`.cfe`
- Добавляется **CF-writer** (пересборка контейнера) как самостоятельный компонент проекта
- Парсер скобочного формата метаданных 1С реализуется на Rust

## 12. Глоссарий

| Термин | Определение |
|--------|-------------|
| **Артефакт** | Бинарный файл платформы 1С (.cf, .epf, .erf, .cfe, .1cd) |
| **Контейнер** | Файл в формате CF, содержащий иерархическую структуру блоков |
| **Виртуальная ФС** | Представление содержимого контейнера в виде дерева файлов/папок |
| **Far Plugin API** | C-совместимый интерфейс FAR Manager для написания нативных плагинов |
| **C ABI** | Application Binary Interface — соглашение о вызовах для совместимости Rust ↔ C |
| **BSL** | Язык программирования платформы 1С:Предприятие |
| **CF** | Файл конфигурации 1С (Config File) |
| **EPF/ERF** | Внешняя обработка / внешний отчёт |
| **CFE** | Файл расширения конфигурации |
| **1CD** | Файловая база данных 1С |
| **Saby-style** | Структура распаковки, совместимая с EDT/Saby/v8unpack |
| **Lazy loading** | Ленивая загрузка: чтение данных только при обращении |
| **Каскадная сборка** | Автоматическое обновление родительского контейнера при изменении вложенного |
