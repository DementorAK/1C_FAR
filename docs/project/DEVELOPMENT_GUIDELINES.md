# Руководство по разработке плагина (Development Guidelines)

Этот документ фиксирует ключевые архитектурные решения, ограничения и "подводные камни", выявленные в процессе разработки плагина под разные версии FAR Manager.

## 1. Источники истины (SDK)

**КРИТИЧЕСКИ ВАЖНО:** Запрещено искать заголовочные файлы и определения структур в интернете (на GitHub и т.д.). Различные форки и версии FAR Manager могут иметь несовместимые API.

Всегда необходимо опираться **исключительно на локальные оффлайн SDK**, расположенные в этом репозитории:

*   **`docs/FAR2_PluginSDK`** — использовать для сборки под Linux/macOS (`far2l` / `far2m`).
*   **`docs/FAR3_PluginSDK`** — использовать для сборки под Windows (современный Far Manager 3).

## 2. Особенности API для far2l / far2m (Linux)

Несмотря на то, что `far2l` работает на 64-битных системах (x64) под Linux, его API имеет ряд жестких и неочевидных ограничений, унаследованных от старого C-кода и адаптации под *nix:

### Выравнивание структур (Memory Alignment)
*   **Правило:** Все структуры, взаимодействующие с API FAR, **должны** иметь атрибут `#[repr(C, packed(2))]`.
*   **Причина:** Заголовочные файлы SDK (например, `farplug-wide.h`) используют `#pragma pack(2)`. Если в Rust-коде использовать стандартное выравнивание `#[repr(C)]`, это приведет к смещению полей (особенно указателей на функции в `PluginStartupInfo`) и немедленному падению плагина (`SIGSEGV` / `Signal 0B`) при попытке загрузки плагина в `far2m`.

### Структура `FarDialogItem`
*   **Правило:** В `far2l` структура `FarDialogItem` **не содержит** встроенного массива данных `Data[512]`, как это было в классическом FAR 2.0 для Windows.
*   **Реализация:** Вместо этого используются два поля: указатель `PtrData: *const WCHAR` и размер `MaxLen: usize`.
*   **Инициализация:** При создании элементов диалога (например, `DI_TEXT`, `DI_CHECKBOX`) необходимо инициализировать `PtrData` указателем на широкосимвольную строку, а `MaxLen` устанавливать в `0` (если длина не ограничена). Установка значения текстового поля через `DM_SETTEXTPTR` вместо инициализации `PtrData` может быть не всегда надежна на этапе `DialogInit`.

### Широкие символы (`WCHAR`)
*   **Правило:** Тип `WCHAR` (`wchar_t`) имеет разный размер на разных платформах.
*   **Windows:** `WCHAR` — это 16-битный символ (`u16`, UTF-16).
*   **Linux / macOS:** `WCHAR` — это 32-битный символ (`u32`, UTF-32).
*   В коде необходимо использовать абстракции (например, `string_utils::to_wide`), которые корректно преобразуют Rust `String` в нужный формат нужной размерности с добавлением нуль-терминатора.

## 3. Отладка и краши

*   Если плагин падает при **загрузке** (сразу после запуска FAR) — скорее всего, проблема в выравнивании структуры `PluginStartupInfo` или `PluginInfo`. Проверьте наличие `packed(2)`.
*   Если плагин падает при **открытии диалога** или вызове конкретной функции — проблема в макете структур, передаваемых в функцию (например, неверный размер `FarDialogItem`), или в передаче невалидных/невыровненных указателей.
